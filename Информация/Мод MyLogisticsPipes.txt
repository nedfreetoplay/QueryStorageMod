Есть труба логистики(далее Л-труба) и к ней идут модули.
Если в трубу была вставлен модуль, то труба получает соответствующий nbt тег и
меняет название с припиской модуля. Пример: Логистическая труба с модулем "Поддержка"
И при снятие трубы выпадает Л-труба с другим названием.(Сделано это для того чтобы не таскать с собой
дахерища модулей и один комплект труб и не добавлять похожих друг на друга труб)

У модулей есть gui в виде интерфейса(его можно изменять) и gui типа GIT консоли запросов
с поддержкой команд по типу языка запросов SQL с таблицами(Л-трубами), запросами,
макросами(список команд типа "что если") и формами.

Формы это интерфейс который виден как основной при заходе в Л-трубу и его можно настроить под каждую трубу.
Создание формы по принципу пограммы MS Access.
(Когда ты переключишься с формы на консоль, то увидишь все запросы которые ты сделал управляя через форму.)

Временно(Всё временное - постоянно):
У сети должно быть хранилище запросов(Скриптов). Хранилище должно быть как контроллер в AE2
В этом хранилище должны быть диски памяти на которых хранятся все запросы, макросы, формы и прочее.
Должна быть Рабочая станция в которой происходит создание Л-труб и модулей с компонентами к ним.

У мода должна быть высокая оптимизация рендера в котором отсикается рендер труб как только это не требуется.
Совместимость с forge microblocks и AE2 фасады.
Совместимость с JEI.
Совестимость с Forge и Ткань(как по англ не помню)

Так как логистические трубы чаще всего не будут связаны между собой в притык, то передача по ним предметов
будет через котроль исполнения запроса(если по путил Л-трубы попадается нужный предмет для запроса,
то он перенаправляет его в сторону необходимую для исполнения запроса.)

Жидкость будет передаватся как предмет, но генерировать временный предмет который нельзя взять в руки.
Для генерации оболочки хранения жидкости будет потреблятся сырье, которое нужно своевременно подавать в сеть.

Редстоун будет передаватся как оптический сигнал который будет перенаправлятся в нужную сторону для получения.
То есть если в системе нету трубы настроенной на прием редстоуна нужной частоты, то сигнал никуда не пойдет.
(Редстоун сигнал можно(возможно) будет наблюдать на слегка измененной текстуре трубы и естественно сигнал будет
двигатся также как и предметы(всегда по прямой кроме перекрестков где некуда идти)).

У трубы должно быть ограничение на количество предметов перемещаемых внутри. Трубы будут передовать предметы
с разной скоростью в зависимости от ускорителей предметов(в виде блоков или доп модулей).
Предметы могут терять скорость своего перемещения по трубам если двигаются без промежуточных ускорителей
по трубам, то есть предмет может быть запущен с сумашедшей скоростью, но если не будет доп усорителей по
длинной трубе, то предмет может начать двигатся с очень маленькой скоростью.

У мода должен быть ключ для быстрого снятия блоков из мода(Когда хочешь снять очень много труб, то ключ
будет очень необходим, так как без ключа процесс снятия будет длится гараздо дольше).

Для маленьких логистических систем не требуется хранилище запросов, так как Л-трубы способны хранить
в себе небольшой список задач, но если требуется организация больше чем 8 запросов, то потребуется хранилище.

Трубы требуют для своей работы питание в виде энергии RF(CoFH Team), EU(IC2), J(Buildcraft) и другие
энергетические системы.
